## Mission

Use the P0 kernel to design, challenge, repair, and deploy ServerForge into one real Discord server explicitly authorized by its owner.

## Case-study proof

The case study must capture:

1. Authorized server identity and secrets-redacted before-state manifest.
2. Intended audience, workflows, roles, channels, permissions, onboarding, moderation, and observability.
3. Three materially different Discord topology candidates.
4. Deterministic evaluation criteria and promotion decision.
5. Adversarial permission, privilege-escalation, spam, recovery, and operator-usability tests.
6. Repair record for every material challenge finding.
7. Applied promoted configuration and connected ServerForge runtime.
8. One real end-to-end mission through Discord.
9. Before/after evidence, health results, rollback proof, promotion receipt, and capability genome.

## Required inputs before mutation

- Discord server ID
- Explicit confirmation that the user owns or administers the server
- Discord application/bot installed with the minimum required permissions
- Bot token stored outside Git
- Named execution host: EDEN/Cali or another approved environment
- Approval of the topology candidate selected for live application

The recursive closeout requires the new committed proof bundle, promoted Git branch, draft PR, release manifest, and returned Discord message IDs.
